Computer Vision Case Study

SoCalGuessr

A machine learning project that predicts which Southern California city a Street View image came from using transfer learning, PyTorch, and a compact production-friendly inference pipeline.

The project focuses on turning a noisy real-world image classification task into a deployable model that balances accuracy, file-size constraints, and inference speed.

Project Overview

The goal was to build a neural network that could look at a Google Street View-style image and predict the correct Southern California city. The challenge was not only classification accuracy, but also packaging the solution into a lightweight inference workflow suitable for automated grading and constrained deployment.

Problem

Distinguish between visually similar urban and suburban environments using subtle cues like road layout, vegetation, lighting, signage, and overall streetscape texture.

Approach

Use transfer learning with MobileNetV3-Small, then improve prediction quality with multi-view inference instead of relying only on a single resized image.

Outcome

Produce a compact model and inference pipeline that is accurate enough to be competitive while remaining small, fast, and practical to ship.

How It Works

The final system combines a pretrained visual backbone with a custom classifier head and a stronger inference routine that averages predictions across multiple views of the same panorama.

01

Transfer Learning

Start from an ImageNet-pretrained MobileNetV3-Small backbone instead of training from scratch. This gave the model a strong visual prior and made learning more efficient on a relatively small custom dataset.

02

Panoramic Input Strategy

Resize images to 224 × 384 to preserve the wide Street View framing, which retains more geographic context than forcing the images into a square crop.

03

Compact Classifier Head

Feed the pooled backbone features into a lightweight head with one hidden layer of 1024 units, Hardswish activation, dropout, and a final six-class output layer.

04

Multi-View Inference

At prediction time, average logits across six views of each image: the base image, a horizontal flip, left crop, flipped left crop, right crop, and flipped right crop. This made the final prediction more stable and improved validation accuracy.

Model Details

The final architecture was chosen as a deliberate tradeoff: strong enough to learn useful spatial cues, but small enough to fit strict packaging constraints.

Architecture Snapshot

  • Backbone: MobileNetV3-Small
  • Hidden layers in classifier head: 1
  • Units in hidden layer: 1024
  • Total parameters: 1,524,006
  • Output classes: 6

Training Setup

  • Optimizer: AdamW
  • Loss: CrossEntropyLoss with label smoothing 0.05
  • Epochs for final checkpoint: 3
  • Batch size: 128
  • Training time: about 2.75 minutes for the saved run

Results

The strongest measured pipeline used the saved MobileNet checkpoint together with multi-view test-time augmentation. A heavier EfficientNet experiment was explored but ultimately rejected because it validated worse while also increasing compute cost.

Best Measured Validation Accuracy
73.33%

Achieved by combining the best MobileNetV3-Small checkpoint with six-view inference rather than relying on a single forward pass per image.

Training curve from the final saved checkpoint. The model’s empirical risk decreased over training while validation accuracy improved from 54.60% to 72.84%, before the inference-side improvements pushed the final validation pipeline to 73.33%.

What This Demonstrates

For recruiters and hiring teams, this project is a good example of practical machine learning work: not just fitting a model, but making design choices around constraints, experimentation, and deployment.

ML Engineering

I built a training pipeline, checkpoint workflow, and inference script that can run in a constrained environment without depending on training-time code.

Experimentation

I compared architectures, tuned the inference path, and selected the final approach based on measured validation performance rather than model size alone.

Product Thinking

I optimized for the full problem: predictive quality, inference reliability, file size limits, and a final artifact that could actually be deployed and reviewed.

Tools Used

The project was implemented in Python and PyTorch, with torchvision for transfer learning and image transforms, plus scikit-learn utilities for evaluation workflow.
